# 以动态 Letterboxd 列表替代固定 profile 作为主要发现入口

**决策时间**：#R2
**状态**：✅ 已确认
**关联大纲**：[返回大纲](../outline.md)

---

## 📋 背景

### 问题/需求
当前系统对外暴露的是 `gold-standard`、`hidden-gems`、`fresh-picks` 三个 profile，但其本质只是后端写死的评分/热度过滤规则，无法表达 Letterboxd 世界中真实存在的大量榜单、奖项、导演推荐与社区主题列表。

用户明确表示：
- 不需要 profile 概念；
- 需要直接面向“列表”；
- 列表来源应同时覆盖官方榜单与社区高质量列表；
- 列表应尽量动态从 Letterboxd 获取；
- 现有命名与信息架构应整体重写。

### 约束条件
- 当前系统抓取链路主要围绕 `source -> movie slugs -> movie metadata -> deck` 设计，并未抽象“列表目录”和“列表影片关系”；
- Letterboxd 列表数量巨大，质量参差不齐，若直接全部暴露，会导致产品复杂度和抓取成本快速上升；
- 当前后端抓取健壮性不足，500 错误仍需优先修复；
- 不宜让“用户可见入口数量”与“内部可用列表数量”直接绑定。

---

## 🎯 目标

1. 让用户以“选择列表”而不是“选择 profile”的方式进行发现；
2. 支持官方榜单与社区列表混合接入；
3. 为后续的搜索、收藏、过滤、缓存和个性化打下统一数据模型；
4. 避免继续扩展固定 profile 造成信息架构恶化。

---

## 📊 方案对比

| 方案 | 描述 | 优势 | 劣势 | 决策 |
|------|------|------|------|------|
| A | 保留 3 个 profile，只增加更多阈值规则 | 实现简单，改动小 | 语义贫弱，无法表达真实策展，入口会很快失控 | ❌ |
| B | 保留 profile 名称，但底层改为若干列表混合 | 对现有 UI 兼容较好 | 用户仍面对抽象名词，不知道来源，也不利于探索 | ❌ |
| C | 重写为“列表即入口”，用户直接选择 Letterboxd 列表 | 语义直观，和用户心智一致，可扩展性强 | 需要新增列表目录、缓存、筛选和抓取策略 | ✅ |

---

## ✅ 最终决策

### 选定方案
采用方案 C：以动态 Letterboxd 列表替代固定 profile，作为主要发现入口。

### 决策理由
- 用户目标非常明确：想从真实列表中选择，而不是在抽象 profile 中切换；
- 官方奖项榜单、官方排名、社区主题列表、导演/名人推荐和个人兴趣列表，均更适合用“列表”抽象承载；
- “列表即入口”天然支持后续的搜索、收藏、排序、订阅与推荐；
- 继续维护 profile 只会把一堆复杂策展逻辑隐藏在模糊名称后面，长期不可维护。

### 预期效果
- 前端主入口改为列表浏览/搜索/选择；
- 后端从“profile 返回 deck”演进为“列表目录 + 列表详情 + 列表影片 deck”；
- 内部可先接入一小批稳定来源，再逐步扩大动态抓取范围；
- 用户对内容来源的理解更清晰，产品可解释性更高。

---

## ❌ 被否决的方案

### 方案 A：继续扩展固定 profile
- **否决原因**：只能表达简单阈值筛选，无法覆盖真实策展需求。
- **重新考虑条件**：仅在需要兼容旧客户端时，作为临时兼容层保留。

### 方案 B：保留 profile 外壳，底层切换到列表池
- **否决原因**：虽然比阈值规则强，但仍把内容来源藏在抽象命名后，用户体验和产品透明度不足。
- **重新考虑条件**：若未来需要为新手用户额外提供“精选入口”，可作为列表系统之上的二级包装，而非主模型。
