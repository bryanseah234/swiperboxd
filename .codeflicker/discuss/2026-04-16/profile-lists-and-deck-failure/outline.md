# 讨论：推荐列表策略与发现卡片失败排查

> 状态：进行中 | 轮次：R2 | 日期：2026-04-16

## 🔵 当前焦点

- **从 profile 模式重写为“可直接选择 Letterboxd 列表”的列表系统**
- **`/discovery/deck` 返回 500 的最可能根因与修复顺序**
- **动态抓取官方列表与社区列表时，产品与系统边界如何划分**

## ⚪ 待讨论

- [ ] 列表抓取入口：是否先接 Letterboxd 官方 lists 页面，再扩展到用户/主题列表页
- [ ] 列表元数据：标题、作者、简介、标签、影片数、点赞数、更新时间的统一模型
- [ ] 列表选择 UI：直接列出原始列表，还是增加搜索、过滤、收藏
- [ ] 缓存与同步：列表目录刷新频率、单列表影片缓存、失效策略
- [ ] 权限与风控：大规模抓取时的速率限制、代理、回退策略

## ✅ 已确认

- 不再保留 profile 概念，改为“列表”作为用户可见的一等入口
- 列表来源采用混合策略：官方奖项/官方榜单 + 高质量社区列表
- 列表应尽量动态地从 Letterboxd 获取，而不是手工维护固定 profile 名称
- 现有 `gold-standard / hidden-gems / fresh-picks` 命名与信息架构应重写，不作为未来主模型
- 当前轮需要同时推进两条线：排查并修复 500；输出列表系统重构方案

## ❌ 已否决

- 继续沿用“少量固定阈值 profile”作为长期产品模型（无法表达真实策展语义）

## 📁 归档

| 问题 | 结论 | 详情 |
|------|------|------|
| 当前 profile 的实现方式 | 基于后端固定过滤规则，而非真实外部列表 | 代码位于 `src/api/app.py` |
| 前端 profile 来源 | 通过 `/discovery/profiles` 动态拉取 | 前端位于 `src/web/app.js` |
| 500 报错现象 | ingest 先失败，随后 deck 请求返回 500 | 已定位为“异常路径处理不足 + deck 过滤/数据健壮性不足”的高概率组合 |
| ingest 抓取数据形态 | 抓 trending 等 source，再抓单片 metadata，写入 store | provider 位于 `src/api/providers/letterboxd.py` |
